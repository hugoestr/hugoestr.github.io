The singletasking vs. multitasking computer

The simplest computer architecture allows for one program to run at a time. The user has total control over the computer. And this is how the earlier computers were built. Developers use this model when writing codebecause it lends itself for simple solutions that can be easily reasoned.

Computers were expensive, so having computers waiting around when the code was involved in Input/Output (I/O) was considered a waste. This was particularly important at a time when computing was charged by the instruction executed. So by the 1960s you had groups of people working on creating a multitasking system that would allow programs to use the processor while some time consuming tasks was happening, such as I/O or waiting for user input.

The most natural solution would be to create a system that could switch between different programs. The computer could execute a few lines of code, and switch to another one and execute a few lines of code for that program as well. Also useful would be to switch to another program while one program was busy with I/O operations. Coming up with this solution would also allow for several users to use the same computer at the same time. Since humans are super slow compared to computers, by quickly switching between programs it could give users the illusion that they were the only users controlling the computer.

Each of these program running by their own, mostly ignorant of what other processes in the computer were doing, eventually were called a process.

So to implement the idea successfully, we need at the very least save the location where each program is located, and make sure that we are not using the memory locations that another program is using to prevent the data being overriden. Providing processes and solving these problems is one of the main tasks of modern operating systems.
